【調査・記述における基本ルール】
- 根拠の明示：各主張に対して、根拠となるファイルパス、および関数/クラス名を必ず記載せよ。
- 事実と推論の区別：コードやドキュメントの「明示的記述」と、そこからの「推論」を明確に区別して記述せよ。
- 証拠の不在：該当する証拠や記述が見つからない場合は、推測で補わず「未確認」と明記せよ。
- 未対応理由の深掘り：未対応機能については、「単に実装が見当たらない（未実装）」のか、「設計や前提アーキテクチャ上、対象外となっている」のかを区別して記述せよ。
- 実際に動かして検証する際は，`uv.lock` を参照するように，`uv run` を用いて検証せよ．

以下の項目に沿って「シンドロームグラフからマッチングを出力する過程」に絞って調査せよ：

1. 対象となるグラフ構造と実装範囲
- この repo はシンドロームグラフ（または Decoding graph / Detection graph）上でのマッチング処理を実装またはラップしているか？（未対応なら以降の調査は不要）
- どのような表面符号のグラフ構造を前提またはサポートしているか（Rotated planar の XXZZ / XZZX 由来のグラフ、Toric code 由来の周期的グラフなど）。
- マッチング対象となるグラフの制約を明示せよ（2D限定か、3D時空間グラフに対応しているか、境界ノード（Boundary node）の扱いはどうなっているか等）。
- グラフ・マッチング機能の対応範囲を、以下の項目を列とした Capability Matrix（表）として整理せよ：
  `Graph dimension (2D/3D)` / `Boundary node support` / `Weighted edges support` / `Hyperedge support (for correlated/Y errors)` / `Dynamic graph generation` / `Parallel matching support`

2. グラフ構築とエッジ重みの計算（ノイズモデルの反映）
シンドロームグラフがどのように構築され、ノイズモデルがエッジの重みや構造にどう反映されているかを調査せよ。
- Code capacity (2D) と Phenomenological/Circuit-level (3D / 測定誤りあり) の違いが、グラフのノード生成やエッジ接続にどう反映されているか。
- エッジの重み（コスト）はどのように計算・保持されているか（対数確率、整数値への丸めなど）。
- 相関誤り（Y誤りなど）に対して、グラフを完全に独立したXとZのグラフに分離しているか、あるいはハイパーエッジ等を許容してマッチングを行っているか。

3. マッチングアルゴリズムの概要
- グラフ上でマッチングを解くためのコアアルゴリズム（MWPM, Union-Find, 貪欲法, テンソルネットワーク等）を特定せよ。
- 外部ライブラリ（PyMatching, Blossom V など）への依存があるか、あるいはスクラッチでの独自実装か。
- ※注意：物理的なQECの文脈ではなく、「グラフ理論・アルゴリズムの実装」として、コードと docstring / README から確認できる範囲でアルゴリズムの挙動を記述せよ。

4. 入出力インターフェースとデータ構造
「シンドロームグラフと欠陥（Defect）の入力」から「マッチング結果の出力」までのパイプラインについて、以下を明確にせよ。
- 入力データ（グラフ）：シンドロームグラフ自体はどのようなデータ構造で表現されているか（NetworkX, 独自の隣接リスト, CSR行列など）。
- 入力データ（シンドローム）：マッチングの対象となる欠陥ノード（Detection events）は、ソルバに対してどのように渡されるか（ノードIDのリスト、バイナリ配列など）。
- 出力データ：マッチングの結果はどのような形式で返されるか（マッチングされたノードIDのペアリスト、オンになったエッジのリスト、物理エラーのバイナリ文字列など）。
- コアとなる関数・クラスのシグネチャ（引数と戻り値の型）を明記せよ。

5. Neural network 系アルゴリズムの適用（該当する場合のみ）
- グラフのエッジ重みの推論（Syndrome decoding）や、マッチングエッジの直接予測（GNNなどによる分類）にニューラルネットワークが使われているか。
- 使われている場合、グラフやシンドロームのデータ構造をどのようにテンソルに変換して入力しているか。

6. マッチング処理のパフォーマンス・ベンチマーク
- ベンチマーク結果や評価スクリプトが存在する場合、QEC全体の論理エラー率ではなく、「マッチングソルバとしての性能」を評価している箇所があるか。
- 何を評価指標としているか明示せよ（マッチングの実行時間、メモリ使用量、グラフサイズやシンドローム密度に対するスケーリング、厳密解とのコスト差など）。
